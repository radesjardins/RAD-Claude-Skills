# Dockerfile Scaffold

Generate production-ready Docker files for the current project. Detect the project type, gather configuration details, generate all Docker files with explanatory comments, and confirm with the user before writing.

---

## Phase 1: Detect Project Type and Configuration

1. Read `package.json` to determine:
   - `name`, `version`
   - Presence of `next` in dependencies → Next.js
   - Presence of `@nestjs/core` → NestJS
   - Otherwise → Express/generic Node.js
   - The `build` script (to know the compile command)
   - The `start` script (to know the runtime command)
   - The `main` entry point or `dist/` output

2. Check `next.config.js` or `next.config.ts` if it exists:
   - Look for `output: 'standalone'`
   - If not present, plan to add it and note it to the user

3. Check for TypeScript:
   - Look for `tsconfig.json` — if found, project is TypeScript
   - Determine output directory from `tsconfig.json` `outDir` (commonly `dist/`)

4. Check for existing `Dockerfile` or `.dockerignore` — warn the user before overwriting

5. Determine the application port:
   - Search for `PORT` in `package.json` scripts, `.env.example`, `src/` files
   - Default to 3000 if not found

6. Check for private npm packages:
   - Look for `@org/` scoped packages in `package.json` dependencies
   - Look for existing `.npmrc` file
   - If found, note that BuildKit secrets will be needed and include the commented pattern

## Phase 3: Generate Dockerfile

Generate the appropriate Dockerfile based on detected project type.

### For Next.js Projects

Check if `output: 'standalone'` is present in `next.config.js`. If not:
- Tell the user: "I need to add `output: 'standalone'` to your `next.config.js` for the optimal Docker build. This enables Next.js to create a self-contained server without copying all of `node_modules`. May I add it?"
- If approved, edit `next.config.js` to add the setting

Generate Dockerfile using the Next.js standalone pattern:

```dockerfile
# syntax=docker/dockerfile:1
#
# Production Dockerfile — Next.js (standalone output)
# Generated by rad-docker

FROM node:22-alpine AS deps
WORKDIR /app
RUN apk add --no-cache libc6-compat
COPY package.json package-lock.json ./
# For private npm packages, uncomment:
# RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm ci
RUN npm ci

FROM node:22-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
ENV NEXT_TELEMETRY_DISABLED=1
RUN npm run build

FROM node:22-alpine AS runtime
RUN apk add --no-cache dumb-init
WORKDIR /app
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1
ENV PORT=3000
ENV HOSTNAME=0.0.0.0
COPY --from=builder --chown=node:node /app/.next/standalone ./
COPY --from=builder --chown=node:node /app/.next/static ./.next/static
COPY --from=builder --chown=node:node /app/public ./public
USER node
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=10s --retries=3 --start-period=60s \
  CMD wget -qO- http://localhost:3000/api/health || exit 1
ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "server.js"]
```

### For NestJS Projects

```dockerfile
# syntax=docker/dockerfile:1
#
# Production Dockerfile — NestJS
# Generated by rad-docker

FROM node:22-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
# For private npm packages, uncomment:
# RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm ci
RUN npm ci
COPY . .
RUN npm run build
RUN npm prune --production

FROM node:22-alpine AS runtime
RUN apk add --no-cache dumb-init
WORKDIR /app
ENV NODE_ENV=production
ENV PORT=3000
COPY --from=builder --chown=node:node /app/node_modules ./node_modules
COPY --from=builder --chown=node:node /app/dist ./dist
COPY --from=builder --chown=node:node /app/package.json ./package.json
USER node
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=10s --retries=3 --start-period=40s \
  CMD wget -qO- http://localhost:3000/health || exit 1
ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "dist/main.js"]
```

Determine the entry point from `package.json` `main` field or `start` script. Use `dist/server.js` as default for TypeScript projects, `src/server.js` or `index.js` for plain JS.

## Phase 4: Generate .dockerignore

Always generate a comprehensive `.dockerignore`. Use the template from `skills/docker/examples/.dockerignore-nodejs` (path relative to the plugin root, not this skill's directory).

Customize based on the project type:
- Next.js: ensure `.next/` is excluded
- Projects with Prisma: exclude `prisma/*.db` (local SQLite files) but keep `prisma/schema.prisma`
- Projects with test databases: exclude `*.db`, `*.sqlite`

---

## Phase 5: Generate docker-compose.yml (If Requested)

If the user confirms they want a `docker-compose.yml`, generate a minimal but production-correct version:

```yaml
name: [project-name-from-package.json]

services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      NODE_ENV: production
      PORT: 3000
    restart: unless-stopped
    # If database detected, add:
    # depends_on:
    #   db:
    #     condition: service_healthy
```

If the project has database dependencies detected (Prisma, knex, pg, mysql2, mongoose):
- Add a database service with health check
- Add `depends_on` with `condition: service_healthy`
- Document required environment variables

---

## Important Rules

1. **Read before you write.** Always detect project type from actual files before generating.
2. **Never overwrite without warning.** If Dockerfile or .dockerignore already exist, confirm before overwriting.
3. **Adapt to the project.** Use the actual entry point from `package.json`, not hardcoded defaults.
4. **Explain your choices.** Add inline comments to generated Dockerfiles explaining non-obvious decisions.
5. **Offer next steps.** Always end with actionable build and run commands.
