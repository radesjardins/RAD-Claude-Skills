## Key Decisions
- **"Mechanism over marketing" is the unifying pattern across all honesty passes.** Convert aspirational claims into Python validators. Same playbook every plugin: read, research, synthesize, present plan, execute. Five plugins shipped this session prove it scales.
- **Retire rad-stack-guide entirely.** Two reasons converged: (a) its stack-detection value duplicated rad-session's Phase 2.5 Resource Discovery, (b) its review-orchestration value collapsed when the framework reviewers were archived. Absorbed `/init` into rad-session 3.0 instead. Cleaner mental model: one plugin owns the workflow lifecycle (init → startup → wrapup).
- **Keep "rad-session" name; reject rename to rad-manager.** Name recognition + searchability matter more than perfect scope-naming. Adding `/init` doesn't require renaming.
- **Each plugin pass dispatches research BEFORE writing.** Specialized agents (rad-brainstormer:domain-researcher, general-purpose verifier, etc.) do the source-checking in parallel. Saves my context, gives independent verification, surfaces things I'd miss eyeballing the plugin alone. Found: agentic-company-builder's "Agentic Bible 2026" was fabricated; rad-writer's "4-Part Hook Formula" was author's invention; rad-coolify-orchestrator's Railpack-availability claim was wrong; rad-writer's "delve 50% spike" undershot reality (~654%); etc.
- **Major version bumps (2.0 / 3.0) when work is platform-level + adds new mechanism.** Don't stage as 2.0→2.1 if the honesty changes only make sense alongside the mechanism that backs them.
- **Honest scope statements at the top of each rewritten README.** Lead with "What this won't do" / "Read this first" — the honesty contract sets the tone.

## What NOT To Do
- TRIED: `audit-cicd.py` regex `^\s*-?\s*curl\b` to detect curl invocations
  FAILED BECAUSE: YAML `run: curl ...` has `run:` between the dash and `curl`, so the anchored regex never matched
  CORRECT APPROACH: `\bcurl\b` (match curl anywhere on line) — works for both `- curl ...` direct form and YAML `run:` form
- TRIED: `audit-cicd.py` Bearer-token check using only the regex's small capture group `([^\s\"']+)`
  FAILED BECAUSE: GitHub Actions secrets syntax `${{ secrets.COOLIFY_TOKEN }}` has SPACES inside the braces, so the capture stopped at `${{` and the secret-reference detector saw a 3-char string with no match
  CORRECT APPROACH: scan the full line slice from `Bearer` onward (not just the capture) for SECRET_REF patterns
- TRIED: `check-coolify-env.py` flagging any `.env.example` value with secret-shaped name + length ≥30 chars as a hardcoded secret
  FAILED BECAUSE: legitimate `.env.example` connection-string templates (`DATABASE_URL=postgresql://user:password@localhost:5432/db`) match those criteria but are template content
  CORRECT APPROACH: recognize template markers (`user:password`, `your-`, `@localhost`, `@127.0.0.1`, `@host`, `@example`, `changeme`, `placeholder`) and skip if present without an actual token-shaped value
- TRIED: invoking `rad-session:wrapup` from the same Claude Code session that just shipped rad-session 3.0
  FAILED BECAUSE: plugin runtime cached the prior version (2.2.0) — wrapup ran from cache, not the new 3.0
  CORRECT APPROACH: this is fine for the wrapup itself (the 2.2 version still produces correct output), but verify new plugin versions in a fresh session after `/plugin marketplace update` + reinstall — same trap that hit rad-session 2.2 wrapup last session

## Key Insights
- **The "fabricated authority" diagnostic.** rad-agentic-company-builder cited "The Agentic Bible 2026" 8+ times as canonical authority — it doesn't exist. When auditing other plugins, grep for citations to specific external authorities and verify each one is real. Same diagnostic for unsourced statistics (rad-writer had four unsourced multipliers like "foster 10.8x" — direction was right but the precise numbers came from nowhere).
- **The "research before writing" protocol pays off every single time.** Every plugin pass this session surfaced at least one wrong claim that I would have missed eyeballing the plugin alone. The investment in dispatching 2-4 parallel research agents per plugin is justified — it catches the things that need fixing before I propose a plan, not after.
- **The "mechanism over marketing" principle scales.** Five plugins, ~13 validator scripts, all pure stdlib Python 3.8+. Pattern: when a plugin makes a measurement claim, build the deterministic check that backs it; when it makes a structural claim, build the parser that verifies it. This is what differentiates the rad-* marketplace from a pile of markdown that asks Claude to behave well.
- **The 2026 Coolify state has interesting wrinkles.** Nixpacks is in maintenance mode (Railway moved focus to Railpack 2025); Coolify v4 is `v4.0.0-beta.474` and never went stable; Caddy added as experimental alternative proxy at beta.237; shared variables expanded to four scopes (server/team/project/environment) with `{{scope.VAR}}` syntax in beta.471. The plugin needed updates but the plugin's own `coverage-report.md` was a model of honest source-tracking — worth replicating elsewhere.
- **The user's "marketplace tightening" philosophy is durable.** Archived 7 single-framework reviewer plugins between sessions because Opus 4.7 handles those reviews well enough on its own. The bar for keeping a plugin in the marketplace: must add value Opus 4.7 doesn't already provide. Future audits should ask this question first ("would Opus 4.7 do this fine without the plugin?") before investing in mechanism.
- **The wrapup-on-current-version trap repeats.** Just shipped rad-session 3.0 with new `/init` skill, but the live plugin runtime is still serving 2.2.0 from cache. Same trap as last session (rad-session 2.2 wrapup couldn't be tested in-session). Pattern: ship → push → start fresh session → `/plugin marketplace update` → reinstall → THEN test.
